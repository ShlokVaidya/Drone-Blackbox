# 4/24/2026 6 AM - Firmware

_Time spent: 0.75h_

I Programmed a Basic Firmware for the Telemetry Tracker

ESP32 Firmware (telemetry_esp32.ino)
Reads all sensors and sends a CSV packet over SiK every 200ms (5Hz):

- GPS parsed via TinyGPS++ — non-blocking, runs every loop 

- MPU6050 read via raw I2C — accel in g, gyro in deg/s 

- HMC5883L — reads after bypass mode enabled, detects if it's the QMC variant 

- BMP180 via Adafruit library — temp, pressure, barometric altitude

One thing to watch: first time you flash, open Serial Monitor at 115200. It prints [HMC5883L] NOT FOUND or Init OK

```
pip install pyserial rich
python telemetry_viewer.py          # auto-detects SiK USB port
python telemetry_viewer.py --log    # also saves CSV with timestamp
```

With rich installed it shows a live dashboard with three panels — GPS, IMU, Mag+Baro. Without it falls back to plain terminal clear-and-print. Auto-detects SiK radio by looking for CP210x/FTDI in device list. --log saves every packet to a timestamped CSV file for post-analysis.

# 4/24/2026 4 AM - Schematic

_Time spent: 2.883333333333333h_

I moved into KiCad 10 for the PCB layout. This was the part I'd been both looking forward to and dreading.

---
![schematic_page-0001](https://stasis.hackclub-assets.com/images/1777003204850-uswhym.jpg)
---
The passive components — capacitors and resistors — took way longer to sort out than I expected. I had to figure out exactly which footprints to use for my specific parts. The 100nF and 10µF ceramics were 0805 SMD — that's 2.0mm × 1.25mm physically. For hand soldering you always want the _HandSolder variant in KiCad, which has slightly extended pads — C_0805_2012Metric_Pad1.15x1.40mm_HandSolder. The 4.7kΩ resistors were 0603, which is noticeably smaller than 0805, and the footprint is R_0603_1608Metric_Pad0.98x1.07mm_HandSolder.

The 100µF 35V electrolytic was a through-hole radial — both leads come out from the bottom. I'd originally looked at a 4V and then a 6.3V MLCC for this, but both had voltage derating issues at 5V. The robu.in through-hole 35V version was the right call — 7× voltage margin, no derating, simple to solder. For the footprint, it's most likely CP_Radial_D8.0mm_P3.50mm but I need to physically measure the cap before the PCB is finalized because Indian market components vary in body size between batches.

Placement rules I learned the hard way (by putting things in the wrong order and having to redo them): decoupling caps must go right next to the IC's VCC pin — within 2mm ideally. The whole point of a decoupling cap is to act as local charge storage. If it's far away, the inductance of the trace between them defeats the purpose. A cap 5mm away might as well not be there during a fast WiFi transmit spike.

I put the GND copper pour on B.Cu — the back layer — and tied it to the GND net. That single decision cleaned up my routing so much. No more GND traces zigzagging everywhere. Everything just drops a via to the pour.

By the end of this session, I had most components placed. The board isn't routed yet but the schematic ERC was passing clean — no errors, no warnings except a couple of intentional no-connect flags on DRDY and the unused SiK flow control pins.

# 4/24/2026 3 AM - Research

_Time spent: 2.583333333333333h_

I spent the morning going through datasheets. That sounds more organised than it actually was. I had like six browser tabs open, a half-drawn block diagram on paper, and I kept second-guessing my pin choices.

The first thing I got wrong: GPS. I'd been planning to use GPIO16 for UART1 RX. Seemed fine — it showed up in the KiCad schematic as a usable pin. But when I cross-checked against the actual ESP32-S3 datasheet, GPIO16 is XTAL_32K_N — the 32kHz crystal input. Its native UART function is U0CTS (flow control for UART0, not UART1 RX). The actual native UART1 RX is GPIO18 (U1RXD), sitting right next to GPIO17 which is U1TXD. So both GPS pins needed to be on 17 and 18 — that's the proper IO MUX P1 path, no GPIO Matrix routing needed. Clean signal, exactly right.

Then I had to figure out the GY-87 in more detail. It's a single module with three separate sensors on it — MPU6050 for gyro and accelerometer, HMC5883L for magnetometer, and BMP180 for pressure and temperature. All connected to a shared I2C bus. The catch? The HMC5883L isn't directly visible on the main I2C bus. It's wired behind the MPU6050's auxiliary I2C port. To read it, you have to write a bypass bit to MPU6050's register 0x37 first. Without that, the magnetometer just doesn't respond and you spend an afternoon thinking the chip is broken.

The module also has two extra pins — FSYNC and DRDY — that I hadn't noticed before. FSYNC is a frame synchronization input for the MPU6050, used in camera rigs and GPS sync applications. For my project: completely unnecessary, but it must be tied to GND — leaving it floating causes undefined behavior. DRDY is the HMC5883L's data ready output. Since I'm polling over I2C anyway, I just put a no-connect on that one in KiCad.

By the end of this session, my schematic was starting to look like an actual schematic. I also learned that the SiK radio's 6-pin DF13 connector has RTS and CTS pins for hardware flow control — but those are optional. For a sensor node sending a CSV string every 100ms, the four essential pins are just 5V, TX, RX, and GND.
